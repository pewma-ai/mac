### Sorpresas
1. **Mermaid carga al revés de lo intuitivo.** El plugin (`docsify-mermaid`) y la librería (`mermaid.min.js`) deben cargarse *antes* del core de Docsify, no después. Además necesita `startOnLoad: false`. Contraintuitivo pero documentado.
2. **Jekyll rompe `_sidebar.md` silenciosamente.** GitHub Pages corre Jekyll por defecto, que ignora archivos con `_` al inicio. Sin `.nojekyll`, el sidebar simplemente no aparece sin ningún error visible.
3. **Docsify concatena líneas en un solo párrafo.** Las líneas contiguas `→ Link1` / `→ Link2` se renderizan como texto corrido. Requieren sintaxis de lista explícita (`- →`).
4. **Las preguntas abiertas del `wish_list.md` ya tenían respuesta.** Lo que empezó como especulación teórica sobre el Playbook se respondió directamente en la práctica al crear `docsify-github-pages.md` y `convenciones-documentacion.md`.

### Aprendizajes
1. **El Playbook es una materialización de la capa de Capacidad.** El proceso Consolidar convierte decisiones repetidas en guías reutilizables. El playbook es exactamente eso pero para conocimiento operativo. Implicación: el Método MaC debería formalizarlo como estructura recomendada.
2. **Probar localmente simulando la estructura de producción evita problemas de subpath.** El truco `ln -sfn docs /tmp/gh-pages/mac` replica exactamente el entorno de GitHub Pages. Implicación: agregarlo al playbook de Docsify como paso obligatorio de verificación.
3. **Los "gotchas" de herramientas son el mejor material para playbooks.** Cada C-XX de esta sesión se convirtió en una lección aprendida en el playbook. El ciclo es: error → corrección → consolidación → playbook. Es MaC en acción.
